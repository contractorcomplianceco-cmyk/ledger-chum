# Production Wiring Checklist

This checklist takes the `release/production` integration branch (a superset of
Phase 1 data-mode, Phase A visual invoice, Phase B AI styling, email
integration, and Phase C customer payments) from "code-complete" to "live." Work
top to bottom; nothing here should be run against a live database or a real
payment gateway until the corresponding secret is in place.

---

## 1. Environment variables & secrets

All variable names are defined (with safe defaults / comments) in
`.env.example`. Copy it to `.env` and fill in real values. **No secrets are
committed** — every value below must be supplied at deploy time.

### App mode
- `VITE_APP_MODE=production` — flips the app out of demo. In production the mock
  fixtures are never consulted; live server functions back every surface.
- `APP_MODE=production` — server-side counterpart (read by `resolveServerMode()`
  for server functions).

### Payments — Authorize.net (Phase C)
- `PAYMENT_PROVIDER=authorizenet`
- `AUTHORIZENET_ENV=sandbox` → set to `production` for live processing.
- `AUTHORIZENET_API_LOGIN_ID` — API Login ID.
- `AUTHORIZENET_TRANSACTION_KEY` — Transaction Key.
- `AUTHORIZENET_CLIENT_KEY` — public Client Key (Accept.js tokenization).
- `AUTHORIZENET_SIGNATURE_KEY` — Signature Key, used to verify settlement
  webhook signatures.

### Email — outbound SMTP (email integration)
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_SECURE` (`true` for 465, `false` for
  STARTTLS/587)
- `SMTP_USERNAME`, `SMTP_PASSWORD`
- `SMTP_FROM_NAME`, `SMTP_FROM_ADDRESS`

### Email — inbound inbox (IMAP/POP)
- `INBOUND_PROTOCOL` — `imap` or `pop3`.
- `INBOUND_HOST`, `INBOUND_PORT`, `INBOUND_SECURE`
- `INBOUND_USERNAME`, `INBOUND_PASSWORD`

### AI invoice styling (Phase B)
- `VITE_INVOICE_AI_PROVIDER` — `local` (default heuristic, no key needed) or a
  hosted provider.
- `INVOICE_AI_API_KEY` / `INVOICE_AI_MODEL` — only required if the provider is
  not `local`.

> Verify there are **no duplicate keys** and no real secrets when you diff your
> `.env` against `.env.example`.

## 2. Database migrations & Supabase types

Do **not** run these against the live DB until reviewed. All migrations are
present and ordered:

- `...phase_c_customer_payments.sql` — `ALTER payments` + `CREATE payment_events`.
- `...a1b2c3d4-...sql` — `CREATE invoice_brand_styles` (Phase B).
- `...email_settings.sql` — `CREATE email_settings` (email integration).
- (`payments` table itself pre-existed in an earlier migration.)

Steps:
1. Review the migration set in `supabase/migrations/` for ordering coherence.
2. Apply migrations to the target environment (staging first).
3. **Re-sync Supabase types** after applying, and confirm the regenerated
   `src/integrations/supabase/types.ts` still contains the union of all new
   tables: `invoice_brand_styles`, `email_settings`, `payments`,
   `payment_events`. (The integrated file already contains all four.)

## 3. Build-time regeneration

- **routeTree**: `src/routeTree.gen.ts` is generated by the TanStack router
  plugin during `vite build`/`dev`. It is committed here for convenience, but a
  clean production build will regenerate it. If routes were added, re-run the
  build and commit the regenerated file.
- **Lockfile**: the repo targets **bun** (`bun.lock`, `bunfig.toml`). Run
  `bun install` in CI/deploy to produce/refresh the lockfile. (This integration
  was verified with `npm` because bun was unavailable; a `package-lock.json` may
  appear locally — do not commit it if the project standard is bun.)

## 4. Payment-receipt → email wiring (ACTION REQUIRED)

The payment-receipt hook is intentionally left as a **clean stub** and is **not**
wired to the email `sendEmail` server function in this branch.

- `src/lib/payments/receipt.server.ts` exposes `registerReceiptSender()` /
  `clearReceiptSender()` and a best-effort `sendPaymentReceipt()`; it currently
  falls back to a dynamic import of `@/integrations/email/send.server`.
- It was left unwired because the coupling is **not trivial or safe**: the email
  `sendEmail` server fn requires a per-org `orgId` plus authenticated context to
  resolve org-specific SMTP config, while the receipt `EmailSender` seam carries
  no org context. A global mutable sender closing over `orgId` is a concurrency
  hazard across orgs, and changing the seam signature would break its contract
  and tests. Receipts are already best-effort and non-blocking, so leaving the
  stub does not affect payment success.

To wire it in production:
1. Extend the `EmailSender` seam (or the receipt call site in
   `src/lib/accounting/customer-payments.functions.ts`, ~line 222) to pass the
   `orgId` + auth context already available in that server function.
2. Register a sender that calls the email `sendEmail` server fn with
   `{ data: { orgId, to, subject, text, html } }`.
3. Keep it best-effort: a receipt send failure must never fail the charge.

## 5. Manual verification steps (post-deploy, staging)

1. **Tokenized card charge** — open an invoice → **Record payment** → gateway →
   card → confirm a successful charge and recorded payment.
2. **Tokenized eCheck / ACH charge** — same flow with an ACH token; confirm
   `pending` (awaiting settlement) status renders correctly.
3. **Settlement webhook** — send/replay an Authorize.net settlement webhook;
   confirm the signature verifies (`AUTHORIZENET_SIGNATURE_KEY`) and the payment
   transitions to settled via `payment_events`.
4. **Send test email** — trigger an outbound email; confirm SMTP delivery from
   the configured `SMTP_FROM_ADDRESS`.
5. **Fetch inbox** — run the inbound fetch; confirm messages arrive via
   IMAP/POP per `INBOUND_PROTOCOL`.
6. **Generate invoice PDF** — open an invoice → **PDF/Print**; confirm the
   client-facing document renders.
7. **Apply AI style** — open the invoice style panel, apply an AI-generated
   theme, and (optionally) save it as the company brand default.

## 6. Guardrails

- Never merge, push to, or open a PR against `main` from this integration work.
- Preserve the superset: every phase's functionality must remain working.
- No secrets in git — supply all values via environment at deploy time.
